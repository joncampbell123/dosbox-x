00369  * there exist old DOS games and demos that rely on very selective IRQ masking,
00370  * but, their code also assumes that calling into DOS or the BIOS will unmask the IRQ.
00371  *
00372  * This fixes "Rebel by Arkham" which masks IRQ 0-7 (PIC port 21h) in a VERY stingy manner!
00373  *
00374  *    Pseudocode (early in demo init):

01829 //
01830 // I don't know exactly how things differ in NEC's PC-98 MS-DOS, but,
01831 // according to some strange code in Touhou Project that's responsible
01832 // for blanking the text layer, there's a "row count" variable at 0x70:0x12
01833 // that holds (number of rows - 1). Leaving that byte value at zero prevents
01834 // the game from clearing the screen (which also exposes the tile data and
01835 // overdraw of the graphics layer). A value of zero instead just causes the
01836 // first text character row to be filled in, not the whole visible text layer.
01837 //
01838 // Pseudocode of the routine:
01839 //
01840 // XOR AX,AX
01841 // MOV ES,AX
01842 // MOV AL,ES:[0712h]            ; AX = BYTE [0x70:0x12] zero extend (ex. 0x18 == 24)
01843 // INC AX                       ; AX++                              (ex. becomes 0x19 == 25)
01844 // MOV DX,AX
01845 // SHL DX,1
01846 // SHL DX,1                     ; DX *= 4
01847 // ADD DX,AX                    ; DX += AX     equiv. DX = AX * 5
01848 // MOV CL,4h
01849 // SHL DX,CL                    ; DX <<= 4     equiv. DX = AX * 0x50  or  DX = AX * 80
01850 // ...
01851 // MOV AX,0A200h
01852 // MOV ES,AX
01853 // MOV AX,(solid black overlay block attribute)
01854 // MOV CX,DX
01855 // REP STOSW
01856 //
01857 // When the routine is done, the graphics layer is obscured by text character cells that
01858 // represent all black (filled in) so that the game can later "punch out" the regions
01859 // of the graphics layer it wants you to see. TH02 relies on this as well to flash the
01860 // screen and open from the center to show the title screen. During gameplay, the text
01861 // layer is used to obscure sprite overdraw when a sprite is partially off-screen as well
01862 // as hidden tile data on the right hand half of the screen that the game read/write
01863 // copies through the GDC pattern/tile registers to make the background. When the text
01864 // layer is not present it's immediately apparent that the sprite renderer makes no attempt
01865 // to clip sprites within the screen, but instead relies on the text overlay to hide the
01866 // overdraw.
01867 //
01868 // this means that on PC-98 one of two things are true. either:
01869 //  - NEC's variation of MS-DOS loads the base kernel higher up (perhaps at 0x80:0x00?)
01870 //    and the BIOS data area lies from 0x40:00 to 0x7F:00
01871 //
01872 //    or
01873 //
01874 //  - NEC's variation loads at 0x70:0x00 (same as IBM PC MS-DOS) and Touhou Project
01875 //    is dead guilty of reaching directly into MS-DOS kernel memory to read
01876 //    internal variables it shouldn't be reading directly!
